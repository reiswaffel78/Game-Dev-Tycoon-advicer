# PROMPT FOR REPLIT AGENT

You are working in an existing web app codebase. Implement language-aware routing + SEO-friendly i18n URLs with these strict rules:

## 1) URL / Routing Rules (must match exactly)

- Default language is **English (en)**.
- **English stays on the base route (no language prefix):**
  - `/<page>` (example: `/page`) = English (default)
- **All other languages must be served under a language subpath:**
  - `/<page>/<lang>/` (examples: `/page/de/`, `/page/fr/`, `/page/es/`)
- **English must also be accessible as an explicit language path:**
  - `/page/en/` should work, but to avoid duplicates do **one** of these (prefer A):
    - **A (preferred):** 301 redirect `/page/en/` → `/page`
    - **B:** serve `/page/en/` with canonical pointing to `/page`
- Enforce consistent trailing slashes:
  - `/page/de` → **301** → `/page/de/`
  - `/page/fr` → **301** → `/page/fr/`
  - Keep the English base route as `/page` (no trailing slash). If `/page/` exists, redirect it to `/page`.

Supported locales: `en`, `de`, `fr` (make it easy to add more later).

## 2) i18n Content Handling

- Implement a clean i18n system (translation dictionaries or i18n library appropriate for this stack).
- Store translations in structured files (e.g., `locales/en.json`, `locales/de.json`, `locales/fr.json`).
- Provide a simple function/helper like `t(key)` to fetch localized strings.
- Fallback behavior:
  - If a key is missing in a locale → fallback to English.
  - If an unsupported locale is requested → 404 or redirect to English base route (choose the most SEO-safe option for this framework; prefer 404 for unknown locales, or redirect to `/page` if that’s standard in the app).

## 3) Language Switcher

- Implement/adjust a language switch UI that:
  - Switches the URL correctly while keeping the same page context.
  - Examples:
    - From `/page` switching to German → `/page/de/`
    - From `/page/de/` switching to French → `/page/fr/`
    - From `/page/fr/` switching to English → `/page`
- Persist user choice in a cookie (preferred) or localStorage.

## 4) SEO Requirements (must be correct per language URL)

For every language version of a page:

- Set `<html lang="...">` correctly.
- Set a **self-referential canonical**:
  - `/page` canonical = `/page`
  - `/page/de/` canonical = `/page/de/`
  - `/page/fr/` canonical = `/page/fr/`
- Add correct `hreflang` alternates:
  - Include `en`, `de`, `fr` and `x-default`.
  - Each page must reference all language variants, including itself.
  - `x-default` must point to the English default route (e.g., `/page`).
- Localize SEO meta per language:
  - `<title>`
  - `meta name="description"`
  - OpenGraph/Twitter (if present)

## 5) Tech/Framework Guidance (do the right thing for the current stack)

- Detect whether this is Next.js / Remix / React Router / Express / etc., and implement using best practices for that framework.
- If Next.js App Router:
  - Use route groups or dynamic segments to support `/<page>/<lang>/` while keeping `/<page>` as default English.
  - Implement redirects via middleware or route config.
- Ensure SSR/SSG output contains correct meta tags server-side (no SEO-only client-side hacks).

## 6) Acceptance Tests (verify these)

- `GET /page` → 200, English content, canonical `/page`, hreflang present, html lang="en"
- `GET /page/en/` → 301 → `/page` (preferred) OR 200 with canonical `/page`
- `GET /page/de/` → 200, German content, canonical `/page/de/`, html lang="de"
- `GET /page/de` → 301 → `/page/de/`
- `GET /page/fr/` → 200, French content, canonical `/page/fr/`, html lang="fr"
- Language switcher updates URL correctly and persists selection.

Deliver the implementation as a clean PR-style change: minimal duplication, reusable helpers, and clear file structure.
